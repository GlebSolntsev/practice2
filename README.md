# Ответы на контрольные вопросы

1.	На какие три группы делятся шаблоны GoF? Приведите по 2–3 примера из каждой группы.

Порождающие (Creational) — отвечают за механизмы создания объектов, позволяя сделать систему независимой от способа создания и композиции объектов.
Примеры: Singleton (Одиночка), Factory Method (Фабричный метод), Abstract Factory (Абстрактная фабрика).

Структурные (Structural) — определяют, как из классов и объектов образуются более крупные структуры, обеспечивая их гибкость и эффективность.
Примеры: Adapter (Адаптер), Decorator (Декоратор), Facade (Фасад).

Поведенческие (Behavioral) — заботятся об эффективной коммуникации между объектами и распределении обязанностей между ними.
Примеры: Observer (Наблюдатель), Strategy (Стратегия), Command (Команда).

2.	Что такое паттерн Strategy? Чем он отличается от State? Как реализовать Strategy в Go через интерфейсы?

Паттерн Strategy позволяет вынести набор алгоритмов в отдельные классы и менять их прямо во время выполнения программы. Это избавляет от громоздких конструкций if/else или switch, когда нужно выбрать один из нескольких способов действия.

Strategy: Используется для выбора как выполнить конкретное действие (например, способ оплаты: картой или наличными). Объекты стратегий обычно не знают друг о друге, а выбор делает клиент.
State: Используется для изменения поведения объекта в зависимости от его состояния (например, заказ: «Новый» -> «Оплачен» -> «Доставлен»). Состояния часто сами знают, в какое следующее состояние должен перейти объект.

Для реализации Strategy в Go нужно выполнить три шага:
1. Создать interface, который описывает сигнатуру метода (алгоритма). Например, type RouteStrategy interface { BuildRoute(a, b string) }.
2. Реализовать этот интерфейс в разных структурах. Одна структура может строить маршрут для автомобиля, другая — для пешехода. В Go не нужно явно писать implements, достаточно просто создать метод с такой же сигнатурой.
3. Создать структуру-контекст, которая содержит поле с типом  интерфейса. В этой структуре должен быть метод для смены стратегии (сеттер) и метод для запуска алгоритма, который просто вызывает метод интерфейса.

3.	Опишите роли Subject и Observer в паттерне Observer. Приведите пример из вашей работы. Как можно реализовать Observer на Go с использованием каналов?

В паттерне Observer взаимодействие строится вокруг двух главных ролей:
Subject (Субъект/Издатель) - это объект, обладающий важным состоянием (например, статус курса или оценка за тест). Он хранит список всех подписанных на него наблюдателей и рассылает им уведомления, когда в его «жизни» что-то меняется.
Observer (Наблюдатель/Подписчик) - это интерфейс объекта, который хочет получать новости. Когда Субъект присылает сигнал, Наблюдатель реагирует на него (обновляет интерфейс, отправляет письмо или пуш-уведомление).

Пример:
Subject: Как только преподаватель выставляет оценку в базу, этот объект «кричит» всем подписчикам.
Observers: Сервис Email-рассылки (отправляет письмо студенту), сервис Push-уведомлений (шлет сигнал в мобильное приложение) и сервис Генерации сертификатов (если оценка проходная, он начинает сборку PDF).

В Go паттерн Observer часто реализуется через каналы, так как они идеально подходят для передачи событий между потоками. Субъект хранит слайс каналов. Когда происходит событие, он проходит циклом по слайсу и отправляет данные в каждый канал.

4.	В чём отличие Factory Method от Abstract Factory? Как отсутствие наследования в Go влияет на реализацию этих паттернов?

Factory Method - это один метод, который создает один тип продукта (например, только отчеты). Вы переопределяете этот метод в подтипах, чтобы изменить формат (PDF или HTML). Это уровень одного метода.
Abstract Factory - это целый интерфейс («фабрика фабрик»), который создает семейства связанных объектов. Например, фабрика для темы оформления: она создает сразу и кнопку, и чекбокс, и окно в едином стиле (Dark или Light). Это уровень объекта-группировки.

В классических языках (Java, C++) эти паттерны строятся на жесткой иерархии классов и extend. В Go наследования нет, есть встраивание (embedding) и интерфейсы.
В Go мы не наследуем логику «родителя». Мы просто реализуем методы интерфейса. Если структура совпадает с интерфейсом — она ему подходит.
Вместо того чтобы «быть» наследником фабрики, структура в Go может включать в себя другую структуру. Это делает код чище, мы не привязаны к дереву наследования и можем легко комбинировать разные поведения.
В Go паттерны выглядят более «плоскими». Нам не нужно строить пятиэтажные цепочки классов, чтобы просто заменить один метод создания. Достаточно передать нужную структуру, реализующую интерфейс.

5.	Что такое «открыть для расширения, закрыть для изменения» (Open/Closed Principle)? Какие паттерны помогают ему следовать? Как этот принцип реализуется в Go через интерфейсы?

Принцип Open/Closed (OCP) — это правило архитектуры, которое гласит: вы должны иметь возможность добавлять в систему новые функции, не трогая при этом старый, уже работающий код.

Strategy: Чтобы добавить новый способ оплаты, ты не правишь метод Pay(), а создаешь новую стратегию.
Decorator: Позволяет «навешивать» новые функции на объект (например, логирование), не меняя сам объект.
Observer: Можно добавить сколько угодно новых подписчиков, и издатель об этом не узнает.
Factory Method: Чтобы научить систему создавать новый тип отчетов, достаточно добавить новую фабрику, не меняя логику существующих.

В Go этот принцип реализуется благодаря неявным интерфейсам.
Вы описываете интерфейс (контракт) — что объект должен уметь делать.
Основная функция принимает этот интерфейс, а не конкретный тип.
Чтобы расширить систему, вы просто создаете новую структуру, которая реализует те же методы.

6.	Какая разница между композицией и встраиванием (embedding) при реализации Decorator в Go? Приведите пример.

В Go разница между этими подходами в реализации Декоратора заключается в том, как мы обращаемся к методам оборачиваемого объекта и насколько «прозрачно» это выглядит для кода.
1. Композиция (Явное поле):
Мы сохраняем объект внутри структуры под конкретным именем (например, Base). Чтобы вызвать его методы, мы должны обращаться к нему напрямую.
мы полностью контролируем вызов. Если мы не переопределим метод в декораторе, он не будет доступен снаружи через структуру декоратора.
2. Встраивание (Embedding / Анонимное поле):
Мы вставляем интерфейс или структуру без имени поля. Все методы вложенного объекта автоматически «всплывают» (promote) на уровень декоратора.
Если мы не переопределили метод, декоратор будет вести себя точно так же, как оригинал. Это позволяет реализовать только один нужный метод, оставив остальные «как есть».

7.	Почему Singleton тяжело тестировать? Как реализовать потокобезопасный Singleton в Go с помощью sync.Once? Какая альтернатива Singleton существует в Go?

Главная проблема тестирования — это скрытая зависимость и общее состояние.
Если один тест изменил настройки Синглтона (например, адрес БД), то все остальные тесты, запущенные следом, «увидят» это изменение. Это делает тесты зависимыми друг от друга.
Синглтон обычно жестко прописан в коде через GetFactory(). Мы не можем легко подсунуть «фальшивый» объект для тестов, не переписывая сам код.
Тесты в Go часто бегают одновременно. Если они все обращаются к одному Синглтону, возникают гонки данных или непредсказуемое поведение.

Потокобезопасный Singleton в Go через sync.Once:

package main

import (
    "sync"
)

type Database struct {
    connection string
}

var (
    instance *Database
    once     sync.Once
)

func GetDatabase() *Database {
    // once.Do сама блокирует другие потоки, пока первый создает объект
    once.Do(func() {
        instance = &Database{connection: "Connected to DB"}
    })
    return instance
}

Альтернатива - это Dependency Injection (Внедрение зависимостей):
Вместо того чтобы объект сам решал, как себя создать и хранить, мы передаем его туда, где он нужен.

8.	Что такое Facade и чем он отличается от Adapter? Как эти паттерны выглядят в Go-проектах?

Facade создает новый, простой интерфейс поверх огромной и сложной подсистемы. Его цель упростить использование.
Adapter делает так, чтобы объект с одним интерфейсом мог работать там, где ждут другой.Его цель совместить несовместимое.

9.	Как реализован выбранный вами паттерн на Go? Какие особенности языка вы использовали (интерфейсы, встраивание, каналы, sync.Once)?

Фабрика в моём проекте реализована как Singleton. Я использовал пакет sync и его механизм sync.Once, чтобы гарантировать создание единственного экземпляра фабрики даже при высоконагруженных параллельных запросах. Это защищает систему от лишнего расхода памяти и гонок данных (race conditions).

Для пошаговой сборки отчета я использовал структуры и указатели. Чтобы избежать дублирования общего функционала (например, логики хранения итоговой строки), я применил встраивание.

10.	Какая из трёх групп GoF предпочтительнее всего используется в выбранной вами предметной области и почему? Учитывая идиоматичность Go, какие паттерны вы бы заменили на встроенные механизмы языка (например, каналы вместо Observer)?

В моей предметной области "платформы онлайн-обучения" наиболее востребована группа поведенческих паттернов.
Это связано с тем, что обучение — это процесс, состоящий из цепочки событий: студент сдает тест → система проверяет ответы → обновляется статистика → рассылаются уведомления → генерируется сертификат. Поведенческие паттерны (Observer, Strategy, State) идеально описывают эти взаимодействия и реакцию системы на действия пользователя.

11.	Чем отличается неявная реализация интерфейсов в Go от явной в других языках? Как это влияет на применение паттернов проектирования?

Явная (Java/C#). Мы обязаны написать class PDFReport implements IReporter. Если вы забыли это сделать или у вас нет доступа к исходному коду класса (например, это чужая библиотека), мы не сможем использовать этот класс там, где нужен IReporter.
Неявная (Go). Мы просто создаём структуру и методы. Если их названия и сигнатуры совпадают с интерфейсом, Go считает, что структура уже его реализует. Это называется «утиная типизация». Если объект ходит и крякает как утка, то это утка.

12.	Как в Go организовать внедрение зависимостей (Dependency Injection) вместо использования Singleton? Покажите на примере.

Внедрение зависимостей (Dependency Injection) вместо Singleton — это переход от модели «объект сам находит себя в глобальной переменной» к модели «объекту дают всё необходимое снаружи». В Go это реализуется через конструкторы и интерфейсы.

Пример (Сервис уведомлений):
Мы не привязываемся к конкретной реализации, нам важно только поведение.

type Notifier interface {
    Send(userID int, message string) error
}

Сервис не вызывает GetEmailService(), он хранит интерфейс внутри себя.

type CourseService struct {
    notifier Notifier // Зависимость
}

// Конструктор — здесь происходит внедрение (Injection)
func NewCourseService(n Notifier) *CourseService {
    return &CourseService{notifier: n}
}

func (s *CourseService) CompleteCourse(userID int) {
    // Используем внедренную зависимость
    s.notifier.Send(userID, "Поздравляем с окончанием курса!")
}

Вместо того чтобы разбрасывать создание объектов по всему коду, мы делаем это в одном месте.

func main() {
    // 1. Создаем конкретные реализации (низкоуровневые детали)
    emailSrv := &EmailNotifier{APIKey: "12345"}
    
    // 2. Внедряем их в высокоуровневые сервисы
    courseSrv := NewCourseService(emailSrv)

    // 3. Работаем
    courseSrv.CompleteCourse(42)
}
